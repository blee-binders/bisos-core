** Info About Avoiding Password/Browser Prompts on Gitlab by Using SSH Keys instead of https
** Full Instructions are at: https://docs.gitlab.com/user/ssh/
Use SSH keys to communicate with GitLab | GitLab Docs
Understand how to use SSH keys with GitLab for secure authentication and repository access.
** Short version:
*** On Windows create the key pair or use your existing one.
*** Run::  cat ~/.ssh/id_ed25519.pub | clip
*** On gitlab web page to get to SSH Keys. Click on your avitar. Then profile. Preferences. SSH Keys is under your profile-preferences.
*** On gitlab Go to add a key and paste the public key.
*** After that, when you clone, Don't use https. Use Clone with SSH instead.
*** If you use git on multiple machines, either enter the public key of each machine on gitlab website
*** Or scp your id_ed25519 from your pc to redhat server. And then a chmod 400 ~/.ssh/id_ed25519. You will then be using same keys.
*** That is all there is to it. It works for me.

* /[[elisp:(beginning-of-buffer)][|^]]  [[elisp:(blee:menu-sel:navigation:popupMenu)][Ξ]] [[elisp:(delete-other-windows)][|1]]/
* [[elisp:(show-all)][(>]]  _[[elisp:(blee:menu-sel:outline:popupMenu)][±]]_  _[[elisp:(blee:menu-sel:navigation:popupMenu)][Ξ]]_       [[elisp:(outline-show-subtree+toggle)][| *SSH over HTTPS (Port 443) Workaround:* |]]    [[elisp:(org-shifttab)][<)]] E|
####+END
** +
Corporate firewalls often block outbound SSH (port 22) but leave port 443 open.
GitLab runs a second SSH listener on port 443 at ~altssh.gitlab.com~.
The traffic is still SSH protocol — just arriving on the port firewalls trust.

** Step 1: Test the tunnel first

#+begin_src bash
ssh -T -p 443 git@altssh.gitlab.com
#+end_src

If you see ~Welcome to GitLab, @username!~ — the tunnel works and you can proceed.

** Step 2: Configure ~/.ssh/config

Add the following to ~~/.ssh/config~ (create the file if it does not exist):

#+begin_src
Host gitlab.com
  Hostname altssh.gitlab.com
  User git
  Port 443
  IdentityFile ~/.ssh/id_ed25519
#+end_src

After this, ~git clone git@gitlab.com:group/repo.git~ silently uses port 443.
No change to your git commands is needed.

** +
GitHub also supports SSH over port 443.

** Direct test

#+begin_src bash
ssh -T -p 443 git@ssh.github.com
#+end_src

If successful, GitHub prints a welcome/authentication message and confirms the tunnel works.

** ~/.ssh/config entry

#+begin_src
Host github.com
  HostName ssh.github.com
  User git
  Port 443
  IdentityFile ~/.ssh/id_ed25519
#+end_src

After this, the normal clone URL works as-is:

#+begin_src bash
git clone git@github.com:ORG/REPO.git
#+end_src

*** Notes

- This remains SSH authentication, not PAT/HTTPS authentication.
- This helps when port 22 is blocked but port 443 is open.
- If corporate DPI/proxy policies inspect or intercept non-HTTPS traffic on 443, this may still be blocked.

####+BEGIN: blee:bxPanel:foldingSection :outLevel 1 :sep t :title "SSH vs PAT vs GCM Best Practices" :anchor "" :extraInfo ""
* /[[elisp:(beginning-of-buffer)][|^]]  [[elisp:(blee:menu-sel:navigation:popupMenu)][Ξ]] [[elisp:(delete-other-windows)][|1]]/
* [[elisp:(show-all)][(>]]  _[[elisp:(blee:menu-sel:outline:popupMenu)][±]]_  _[[elisp:(blee:menu-sel:navigation:popupMenu)][Ξ]]_       [[elisp:(outline-show-subtree+toggle)][| *SSH vs PAT vs GCM Best Practices:* |]]    [[elisp:(org-shifttab)][<)]] E|
####+END
** +
Use both SSH keys and PATs. They are complementary.

** Recommended daily split

- SSH keys for daily clone/pull/push on trusted developer machines.
- PAT for fine-grained, time-limited access to a specific repo, API use, and automation.
- GCM for HTTPS workflows when SSH is blocked or when SSO/browser-based sign-in is required.

** Current practice (2026)

- SSH keys are still a best practice for interactive developer workflows.
- PAT usage has increased for automation and scoped delegation, not as a replacement for SSH.
- GCM is a convenience layer for HTTPS credentials storage and token refresh, especially on Windows.

** PAT hygiene

- Always set an expiration date.
- Use minimum scopes (example: read-only when push is not needed).
- Store PAT in credential manager or password manager, never in repo files.
- Rotate/revoke tokens quickly if exposure is suspected.

** Team policy suggestion

- Default to SSH keys for normal development.
- Use PAT only when scoped or temporary access is needed.
- Keep HTTPS+GCM as fallback when corporate network restrictions block SSH.
